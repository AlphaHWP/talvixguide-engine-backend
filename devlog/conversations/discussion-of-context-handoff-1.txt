Talvixguide Engine – Project Context Summary

The project aims to build a high-performance, decentralized media discovery backend using modern C++ standards and cryptographic verification. Below is a structured summary of the core components and roadmap.
Core Design Choices
Language & Concurrency

    C++23 → C++26: Current development uses C++23 (-std=c++23), with plans to adopt C++26 features like std::execution (via NVIDIA's stdexec).

    Concurrency Model: Based on senders/receivers framework, enabling composable asynchronous operations. Example patterns include schedule() | then() and bulk() for parallel execution across CPU/GPU contexts. 

Metadata Formats & Parsing

    TOML: Used for human-authored metadata; parsed via toml++, a header-only, modern C++ library with strong performance and safety.

    JSON: For high-speed machine ingestion; handled by yyjson, known for ultra-fast parsing.

    Both libraries support round-trip formatting and integration with CMake via CPM.cmake.

toml++ vs yyjson performance
Cryptography

    Signing Options:

        OpenSSL: Supports RSA/ECDSA; mature but requires careful configuration.

        Libsodium (Ed25519): Preferred for better performance and security (faster verification, resistance to side-channel attacks). 

    Trust Model: Explicit key trust (out-of-band), with optional JWKS support planned.

    Signature Scope: Supports both file-wide and per-entry signatures.

    🔍 Performance Insight: Libsodium’s Ed25519 outperforms OpenSSL’s ECDSA P-256 in both signing (~1.4x faster) and verification (~3.7x faster), making it ideal for scalable content signing.

    Build System: CMake with embedded CPM.cmake for reproducible dependency management.

    Dependencies:

        toml++ – TOML parsing

        yyjson – High-performance JSON

        stdexec – Structured concurrency (experimental C++26)

        OpenSSL or libsodium – Cryptographic operations

    Compiler Flags: -std=c++23 (current), transitioning to -std=c++2c for C++26 features.

GUI Framework Decision – Talvixguide Engine

Based on the evaluation and current context, Qt remains the recommended choice for the signer and extractor tools.
Key Insights from Research

    Qt is widely regarded as the most mature, feature-rich, and supported C++ GUI framework, especially for production applications requiring native look, cross-platform support, and robust tooling (e.g., Qt Creator, QML).

    Dear ImGui excels in developer tools, debug interfaces, and real-time applications but lacks accessibility, internationalization, and traditional UI scalability.

    wxWidgets has significant drawbacks: limited maintenance, outdated native control support (e.g., dark mode issues on Windows), and a small ecosystem, making it risky for long-term projects. 

    ✅ Consensus: Qt is the "least bad" option for cross-platform desktop apps, despite its MOC preprocessor and larger binary size. 

Recommendation

Proceed with Qt for both the signer and extractor GUI tools. It best meets the project’s needs for:

    Native UX across platforms

    Long-term maintainability

    Developer productivity (IDE, QML, documentation)

    Integration with CMake and modern C++ 

Avoid wxWidgets due to high bus factor and platform inconsistencies. Reserve Dear ImGui for internal dev tools (e.g., debug dashboards).

Qt LGPL-3.0 Compliance & Integration – Talvixguide Engine
Licensing Compliance

    Dynamic linking is the safest path for LGPL-3.0 compliance:

        ✅ Distribute Qt dynamically (e.g., via system install or AppImage with bundled Qt)

        ✅ Allow users to replace Qt libraries (e.g., via $PATH, LD_LIBRARY_PATH, or AppImage extraction)

        ✅ Include LICENSE.LGPL-3.0 and ATTRIBUTIONS.md with copyright notices and source link: https://code.qt.io

    Static linking requires additional obligations (object files, relinking instructions), so it's not recommended. 

Qt Distribution via AppImage

    AppImage is compliant and recommended:

        Bundles Qt and dependencies, ensuring portability

        Users can extract and replace Qt libraries if needed

        Use appimage-builder or linuxdeployqt for CMake integration

        Ensure AppRun sets QT_PLUGIN_PATH and LD_LIBRARY_PATH correctly 

Qt AppImage distribution LGPL compliance
CMake Integration Best Practices

    Modern CMake (3.16+) with Qt6:

    cmake_minimum_required(VERSION 3.16)
    project(talvix-signer LANGUAGES CXX)

    set(CMAKE_CXX_STANDARD 17)
    set(CMAKE_AUTOMOC ON)
    set(CMAKE_AUTORCC ON)
    set(CMAKE_AUTOUIC ON)
    set(CMAKE_INCLUDE_CURRENT_DIR ON)

    find_package(Qt6 COMPONENTS Widgets REQUIRED)
    add_executable(talvix-signer main.cpp mainwindow.cpp mainwindow.h mainwindow.ui)
    target_link_libraries(talvix-signer PRIVATE Qt6::Widgets)
     

    Set CMAKE_PREFIX_PATH to Qt’s lib/cmake during build (e.g., -D CMAKE_PREFIX_PATH=/path/to/Qt/6.5.2/gcc_64/lib/cmake)

# Core Design Context (Egress)

## Vision & Scope
High-performance, decentralized media discovery backend enabling creators to publish cryptographically signed metadata. Users discover content via scalable, pluggable recommendation algorithms.

## Major Components & Responsibilities
- **Backend Engine**: Core ingestion, parsing, indexing (C++23/26)
- **Signer Tool**: Creator-facing GUI/CLI for signing TOML metadata (Qt6)
- **Extractor Tool**: Pulls metadata from platforms (e.g., YouTube API) → TOML
- **Frontend Clients**: User-facing discovery apps (future)
- **Metadata Store**: `/metadata/raw_toml`, `/metadata/processed_json`

## Component Interactions & Data Flow
1. **Creator Workflow**: Extractor → YouTube API → TOML → Signer → Signed TOML → Export
2. **Ingestion Workflow**: Signed TOML → Parse (toml++) → Convert to JSON → Verify (libsodium) → Index (yyjson)
3. **Discovery Workflow**: Indexed JSON → Pluggable recommenders → Frontend

## Technology Stack
- **Language**: C++23 → C++26 (`-std=c++2c`)
- **Concurrency**: NVIDIA `stdexec` (senders/receivers)
- **Metadata**: TOML (creator), JSON (machine)
- **Libraries**: `toml++`, `yyjson`, `stdexec`, `libsodium`/`OpenSSL`
- **GUI**: Qt6 (LGPL-3.0) for signer/extractor
- **Build**: CMake + CPM.cmake

## Architectural Constraints
- **Decentralized Trust**: No central authority; explicit key verification
- **Performance**: Sub-second parsing/verification for large JSON
- **Portability**: Support Linux, Windows, macOS
- **Compliance**: LGPL-3.0 for Qt6 (dynamic linking)
- **Extensibility**: Pluggable recommenders, modular tooling
